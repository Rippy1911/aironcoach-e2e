## P0 BUG — Zły profil po zapisie coacha / My Profile pokazuje inne dane niż tworzę

### Odtworzenie (zgłoszone przez użytkownika)

- Wiele kont testowych na prod
- Po utworzeniu/edycji profilu coacha użytkownik widzi **inny profil** niż ten który edytuje
- My Profile może prowadzić do profilu z danymi innej osoby (np. Marek Nowak) lub starego wiersza testowego
- Create coach profile zapisuje toast sukcesu, ale UserProfile pokazuje coś innego

### Root cause (potwierdzone analizą bundle prod)

**System pozwala na WIELE rekordów `UserProfile` z tym samym `created_by` (email)** — do 50 wierszy:

```js
UserProfile.filter({ created_by: email }, "-updated_date", 50)
```

Następnie funkcja **`uu()`** wybiera „kanoniczny" profil przez **scoring `Fz()`**, który **faworyzuje coacha**:

| Pole | Punkty w Fz() |
|------|----------------|
| is_coach + approved | +1,000,000 |
| coach_onboarding_completed | +200,000 |
| coach_business_name | +100,000 |
| coach_status !== none | +50,000 |
| is_coach | +40,000 |
| onboarding_completed (personal) | +20,000 |
| first_name / last_name | +10,000 / +5,000 |

**Efekt:** jeśli na jednym emailu są 2+ wiersze UserProfile (np. z testów E2E, wielokrotnego onboardingu, create zamiast update):
- `useMyProfile` / menu „My Profile" → zawsze bierze wiersz z **najwyższym score** (prawie zawsze coach)
- `CreateCoachProfileForm` save → `uu(rows)` → **updateuje ten sam „wygrany" wiersz**, niekoniecznie ten który user myśli że edytuje
- Osobny wiersz „personal" zostaje osierocony ze starymi danymi (np. Marek Nowak, test data)
- `/UserProfile?id=<stary_id>` nadal pokazuje stary wiersz — user myśli że to „jego" profil

Dodatkowo istnieje **`bRe()`** które przy wielu coach rows zwraca `status: "ambiguous"` — system **wie** o problemie duplikatów, ale CreateCoachProfile tego nie obsługuje.

**CreateCoachProfile i CreateUserProfile używają tej samej logiki:**
```js
const rows = await UserProfile.filter({ created_by: email }, "-updated_date", 50);
const picked = uu(rows, email);
if (picked?.id) await UserProfile.update(picked.id, payload);
else await UserProfile.create(payload);
```

To jest **fundamentalny bug danych**, nie kwestia UX „coach vs user profile".

## FIX A — Jeden kanoniczny UserProfile na użytkownika (P0)

### Reguła

```
1 user (auth id / email) = max 1 aktywny UserProfile row
Coach capability = pola na TYM SAMYM wierszu, nie osobny rekord
```

### Backend / entity

1. Dodaj unikalność logiczną: `created_by` (email) **lub** `created_by_id` (auth user id) — jeden profil na user
2. Jeśli platforma nie wspiera unique index — backend guard w każdym create:
   - `ensureSingleUserProfile(user)` przed create
   - jeśli istnieje → update, nigdy create drugiego wiersza

### Migration / cleanup (jednorazowo)

Skrypt admin `mergeDuplicateUserProfiles`:
1. Znajdź emaile z `count(UserProfile) > 1`
2. Dla każdego emaila:
   - **Primary** = wiersz z `created_by_id === auth.userId` jeśli jest, inaczej najstarszy z `onboarding_completed`
   - **Merge** z pozostałych: coach fields, billing (premium, credits), name, avatar — do primary
   - **Archive/delete** duplikaty (soft-delete flag `merged_into_id` jeśli delete ryzykowne)
3. Raport: email → merged N rows → primary id

**Priorytet merge coach fields:** jeśli duplikat ma `is_coach: true` a primary nie — przenieś coach pola na primary.

## FIX B — Zastąp uu() / Fz() scoring dla „my profile" (P0)

### Problem

`uu()` z Fz() scoring **nie nadaje się** do wyboru „mojego profilu" — to wybiera „najbardziej coach-like row".

### Nowa logika `getCanonicalUserProfile(rows, user)`

```js
export function getCanonicalUserProfile(rows, user) {
  if (!rows?.length) return null;
  const userId = user?.id;
  const email = user?.email;

  // 1. Exact match by created_by_id (najpewniejsze)
  if (userId) {
    const byId = rows.filter(r => r.created_by_id === userId && !isServiceRow(r));
    if (byId.length === 1) return byId[0];
    if (byId.length > 1) return pickNewest(byId); // + log warning
  }

  // 2. Match by created_by email
  const byEmail = rows.filter(r => r.created_by === email && !isServiceRow(r));
  if (byEmail.length === 1) return byEmail[0];
  if (byEmail.length > 1) {
    // NIE używaj Fz() — preferuj wiersz z OBU: onboarding_completed + najnowszy updated_date
    return pickBestCanonical(byEmail); // see below
  }

  return null;
}

function pickBestCanonical(rows) {
  // Prefer: onboarding_completed true, then max updated_date
  // NIE faworyzuj coach score — coach to pola na tym samym wierszu
  return [...rows].sort((a, b) => {
    const oc = (b.onboarding_completed ? 1 : 0) - (a.onboarding_completed ? 1 : 0);
    if (oc) return oc;
    return new Date(b.updated_date) - new Date(a.updated_date);
  })[0];
}
```

### Gdzie podmienić

- `wRe()` (useMyProfile fetch)
- `CreateCoachProfileForm` save
- `CreateUserProfileForm` save
- `ProfileModeMenu` — `o.id` musi być z canonical profile
- Wszękie `uu(rows, email)` → `getCanonicalUserProfile(rows, user)` z **user object** (id + email)

**Usuń lub ogranicz Fz()** — używaj tylko do rankingu directory, NIE do „which is my profile".

## FIX C — Coach save na tym samym wierszu co personal (P0)

`CreateCoachProfileForm` save:

1. `const profile = await ensureCanonicalProfile(user)` — jeden wiersz, create tylko jeśli zero
2. `update(profile.id, { ...coachFields })` — **nie nadpisuj** first_name/last_name jeśli user nie edytował ich w coach form (albo sync świadomie z osobnej sekcji)
3. Po save: `invalidateQueries(['myProfile', 'userProfile', profile.id])`
4. Redirect: `/UserProfile?id=${profile.id}&tab=coach` (nie losowy id)

**Coach fields na tym samym UserProfile co personal** — zgodnie z unified account. Nie twórz drugiego wiersza „coach profile".
