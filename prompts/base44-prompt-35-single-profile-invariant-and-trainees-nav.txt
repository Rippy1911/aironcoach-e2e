GOAL
═══════════════════════════════════════════════════════════════

Fix the architecture:

  1. Each auth user has exactly one canonical UserProfile.
  2. Coach profile is NOT a second UserProfile.
  3. Coach data/listing details live as fields/extension on the same canonical
     UserProfile.
  4. If the user has a team and is a coach/owner, show Trainees in desktop

═══════════════════════════════════════════════════════════════
CORE INVARIANT: 1 USER = 1 USERPROFILE
═══════════════════════════════════════════════════════════════

There must never be two UserProfile records for the same auth user.

Canonical identity:

  UserProfile.created_by_id === auth.user.id
  OR UserProfile.created_by/email === auth.user.email for legacy rows.

Creating a coach listing must update the existing UserProfile:

  DO:
    update UserProfile set:
      is_coach = true
      coach_status = "pending" / "approved"
      coach_profile_visibility = "private" / "public"
      coach_business_name
      coach_headline
      coach_bio
      specialties
      certifications
      years_experience
      price_per_session
      services metadata, if applicable

  DO NOT:
    create a second UserProfile for the same user
    create a separate "coach profile" row in UserProfile
    link /UserProfile?id=... to a duplicate row

Terminology:
  - UserProfile = one canonical profile per user
  - Coach details / Coach listing = extension fields on UserProfile
  - Services/Offers = separate service records, linked to canonical UserProfile.id

═══════════════════════════════════════════════════════════════
FIX 1 — CENTRALIZE GET-OR-CREATE PROFILE
═══════════════════════════════════════════════════════════════

Create/reuse one helper used everywhere:

  getOrCreateCanonicalUserProfile(authUser)

Logic:

  1. Find UserProfile by created_by_id = authUser.id.
  2. If none, find legacy UserProfile by created_by/email = authUser.email.
  3. If exactly one found:
      - backfill created_by_id if missing/wrong and safe
      - return it
  4. If multiple found:
      - choose canonical:
          newest profile with created_by_id === authUser.id
          else newest profile with created_by/email match
          prefer profile with is_coach=true / coach_status if user has coach data
      - merge duplicates into canonical
      - soft-delete/archive duplicates or mark merged_into_profile_id
      - return canonical
  5. If none:
      - create exactly one UserProfile.

All flows must use this helper:
  - onboarding
  - profile page
  - My Profile link
  - Coach onboarding / create coach listing
  - Coach Hub
  - CoachServices
  - getUserProfile
  - auth/global setup
  - any seed/demo creation for real users

FIX 3 — GETUSERPROFILE RESOLUTION
═══════════════════════════════════════════════════════════════

getUserProfile({ user_id }) must resolve profiles consistently:

  1. user_id in URL is UserProfile.id.
  2. If a legacy User.id is passed, redirect/resolve to canonical UserProfile.id.
  3. If user_id points to archived/merged duplicate:
      - resolve canonical profile
      - include canonical id in response
      - optionally return redirect_profile_id for frontend URL normalization

Response must include:

  id: canonicalProfile.id
  user_id: canonicalProfile.id   // if existing frontend expects user_id
  is_self: true/false
  is_coach
  is_public_coach
  coach_status
  _my_profile_id: canonical own profile id for the caller

is_self logic:

  is_self = caller exists AND canonicalProfile is caller's canonical profile

Do not rely on the requested URL id alone if it may be a merged duplicate.

FIX 6 — TRAINEES NAV ELIGIBILITY
═══════════════════════════════════════════════════════════════

Show Trainees tab/link only when:

  hasTeamAccess && isCoachOrOwner

Define:

  isCoachOrOwner =
    profile.is_coach === true
    OR profile.coach_status === "approved"
    OR user has role "coach"
    OR user is owner/admin of at least one team

  hasTeamAccess =
    user owns a team
    OR user is coach/admin in a TeamMembership

Recommended function:

  canSeeTraineesNav({ user, profile, teams, memberships }) {
    const isCoachOrOwner =
      profile?.is_coach === true ||
      profile?.coach_status === "approved" ||
      teams.some(t => t.created_by_id === user.id || t.owner_user_id === user.id) ||
      memberships.some(m => ["owner", "coach", "admin"].includes(m.role));

    const hasTeamAccess = teams.length > 0 || memberships.length > 0;

    return isCoachOrOwner && hasTeamAccess;
  }

Apply same logic to:
  - desktop sidebar
  - mobile bottom navigation
  - any route guard for /Trainees

Do not show Trainees for regular users with no team.
Do not hide Trainees from approved coaches/team owners on mobile.
